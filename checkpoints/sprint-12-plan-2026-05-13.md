## Orchestrator Critical Thinking Review

### Assumptions Examined

1. **"Sprint 12 can absorb the 24 Sprint 11 carry-overs cleanly."**
   Of the 24 moved issues, several are **duplicates or sub-issues** of items already in Sprint 12:
   - **#93** (ConfigManager migration) overlaps with **#359** (ARCH-005: Consolidate config systems) — same goal, different framing. They should be treated as one work item. #359 is the authoritative version; #93 is the thin-shim removal sub-task.
   - **#281** (Split cli/commands.py) overlaps with **#366** (decompose cli/commands.py god-module) and **#365** (ARCH-004: adopt Click). These are the same underlying problem with different proposed solutions (split vs. adopt Click framework). Architecture Agent must decide which path before dev starts.
   - **#174–177, #218–225** (mono-repo steps) are the Epic + sub-issues for the same restructuring work. They should be implemented sequentially, not in parallel.
   - **Decision:** Treat overlapping issues as a merged work item. #93 → sub-task of #359. #281 → superseded by #366. Epic #174 is parent of sub-issues #218–225.

2. **"53 issues is the Sprint 12 scope."**
   53 issues is the **total milestone backlog**, not the committed sprint scope. The committed scope for a 2-week sprint should not exceed ~20 issues. The PM must explicitly commit P0/P1 issues and select a handful of P2 issues. Everything else stays in backlog within the milestone.

3. **"Sprint 12 can start immediately without QA Phase 1."**
   Several committed issues (#358, #359, #365, #366) are P0/P1 and lack `qa:tests-defined` labels. **The QA Agent must write failing tests for all P0/P1 issues before dev branches are created.** This is non-negotiable per the Development Lifecycle Protocol.

4. **"The mono-repo restructuring (#174–225) can be done this sprint."**
   These are the most disruptive changes in the entire codebase — moving `classes/`, `config/`, `api/`, `cli/`, `services/` to a new `core/`/`app/` structure. Any one of these will break every import in the codebase and require CI gate verification. **Do not commit all 9 mono-repo sub-issues this sprint.** Limit to Step 1 only (#218, #219, #220) and verify CI before proceeding.

5. **"#190 (lab-ci nightly workflow) can be picked up immediately."**
   The CI/CD Agent needs QA Phase 1 tests written first. The acceptance criteria were documented in the issue comment on 2026-05-13. QA Agent must write the tests before CI/CD Agent starts.

### Alternative Perspectives

- **"Skip the mono-repo restructuring entirely and just fix the god-module CLI."** The CLI is 2,348 lines — adopting Click (#365) would be the most user-facing improvement. The risk of the mono-repo restructure is high; the reward is architectural cleanliness. Both are important but the CLI fix has immediate velocity impact. **Counter:** The mono-repo step is a prerequisite for the `v1.0.0` gate (Sprint 13). We must start Step 1 now.
- **"Roll up the duplicate issues (#93 + #359, #281 + #366) before sprint starts."** This is cleaner than having two open issues for the same work. **Decision:** PM Agent should add closing comments to #93 and #281 noting they are superseded, but leave them open until the parent is closed (GitHub will auto-close sub-issues).

### Separate Facts from Opinions

**Facts:**
- Sprint 12 branch `sprint/12` created from `origin/develop` (HEAD commit: Sprint 11 merge) on 2026-05-13
- Sprint 12 has 53 open issues as of 2026-05-13
- P0 issues in Sprint 12: #358 (ARCH-001 layering violation)
- P1 issues in Sprint 12: #359 (config), #365 (CLI Click), #366 (commands decompose)
- No `qa:tests-defined` labels on any P0/P1 Sprint 12 issues
- `develop` branch is fully up to date with Sprint 11 (1693 tests pass)

**Opinions:**
- The mono-repo restructure should be limited to Step 1 this sprint
- #93 and #281 are superseded by newer, more specific issues
- QA Phase 1 is the most urgent action before any dev work starts

### Potential Biases

- **Recency bias:** Sprint 11 success with lab infrastructure may cause over-confidence that architectural changes (mono-repo) will go smoothly. Import path refactors are notoriously fragile — a single missed import breaks all tests.
- **Scope creep risk:** The 24 Sprint 11 carry-overs create pressure to "finally do the big refactor." Resist committing all 9 mono-repo sub-issues. Start with Step 1, verify, then decide on Step 2.
- **Duplicate issue bias:** Having #93 and #359 both open for config migration creates ambiguity about which PR closes which. Resolve before dev starts.

### Risks

| Risk | Likelihood | Impact | Mitigation |
|------|-----------|--------|-----------|
| Mono-repo Step 1 breaks all import chains | High | High | Scope to `classes/` only first; CI gate required on every PR |
| #365 (Click CLI) is an L-size issue, not M | High | Medium | Time-box to 3 days; if blocked, reduce to partial Click adoption on one command |
| #358 (P0 layering violation) is in an untested code path | Medium | High | QA writes regression test first; fix is scope-limited to `draft_setup.py` |
| Duplicate issues (#93/#359, #281/#366) cause double PRs | Medium | Low | PM leaves superseded comment; assign to same dev agent |
| 53-issue milestone creates planning paralysis | Medium | Medium | Commit only 18 issues to sprint; rest are explicitly labeled as backlog |

### One Deeper Question

**Is adopting Click (#365) the right move before the mono-repo restructure, or will the restructure invalidate the Click migration?**
If we adopt Click in `cli/main.py` now, then do the mono-repo move (`cli/ → app/cli/`), we'll need to update all Click entry points AND the package structure simultaneously. **Answer:** The Architecture Agent should decide the sequencing before Sprint 12 dev work starts. Recommended order: mono-repo Step 1 (classes/config) → mono-repo Step 2a–c (services/api/cli) → then Click adoption. This ensures the Click migration targets the final path.

### Possible Consequences

- **If Sprint 12 succeeds (mono-repo Step 1 + config unification + CLI decompose):** The codebase moves materially closer to the `v1.0.0` gate. Sprint 13 (1.0.0 Gate) becomes achievable. The lab pipeline continues to mature.
- **If the mono-repo restructure blocks:** Every other PR in the sprint will conflict. Mitigation: treat the mono-repo PRs as sequential, not concurrent. Only one mono-repo step can be In Progress at a time.
- **If the config unification (#359) conflicts with #93's partial work:** The Sprint 11 work on `CommandProcessor` DI injection is a clean foundation for #359. #93's thin-shim removal is effectively included in the scope of #359. **Treat them as one.**
